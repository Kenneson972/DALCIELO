1. Securite

Analyse des API routes, authentification admin, rate limiting, headers de securite, et protection Stripe

  • ✓ CSP headers complets dans next.config.js
  • ✓ HSTS active en production (1 an + preload)
  • ✓ X-Frame-Options SAMEORIGIN, X-Content-Type-Options nosniff
  • ✓ Validation des prix cote serveur (anti-manipulation)
  • ✓ Rate limiting sur /api/orders (10 req/60s)
  • ✓ Rate limiting sur auth admin (15 fails = lockout 15min)
  • ✓ Stripe webhook signature verification
  • ✓ Endpoint /api/checkout deprecie (410 Gone)
  • ⚠ Rate limiting en memoire (non distribue)
  • ⚠ Pas de CSRF token explicite sur les formulaires
  • ⚠ Admin PIN fallback "1234" en dev
  • ✗ Pas de logging des requetes pour audit trail
Migrer le rate limiting vers une solution distribuee P1
Le rate limiting actuel utilise un Map() en memoire dans src/lib/rateLimit.ts. En cas de deploiement multi-instance (Vercel serverless), chaque instance a sa propre Map, rendant le rate limiting inefficace. Migrer vers Upstash Redis ou Supabase pour centraliser les compteurs IP.
rateLimit.ts kb-security.mdc Production
Ajouter un audit logging sur les endpoints admin P2
Aucun log de securite n'est produit quand un admin valide, modifie ou supprime une commande. Ajouter un middleware de logging qui enregistre : IP, action, timestamp, et resultat dans Supabase (table admin_audit_log).
API routes /admin/* kb-security.mdc
Ajouter une protection CSRF P2
Les formulaires POST (commandes, reviews) n'ont pas de token CSRF. Bien que Same-Origin soit en place, la regle kb-forms.mdc exige un CSRF token. Implementer via un cookie SameSite=Strict + header de verification, ou utiliser le pattern Double Submit Cookie.
CartDrawer.tsx kb-forms.mdc
Configurer CORS explicitement P3
Aucun header CORS n'est configure dans les API routes. La regle kb-backend.mdc exige un whitelist CORS explicite (jamais *). Ajouter Access-Control-Allow-Origin avec le domaine pizzadalcielo.com uniquement.
next.config.js kb-backend.mdc

3. Performance

Lazy loading, optimisation des images, bundle size, re-renders et Core Web Vitals

  • ✓ Dynamic imports pour Chatbot, AnnouncementPopup, StickyCartBar
  • ✓ next/image avec formats AVIF/WebP
  • ✓ Font display: swap sur toutes les polices
  • ✓ Background mobile optimise (webp dedie)
  • ✓ removeConsole en production
  • ✓ Zustand pour le state management (leger)
  • ⚠ 5 polices Google chargees (Inter, Poppins, Montserrat, Indie Flower, Playfair)
  • ⚠ Recharts dans l'admin non lazy-loaded
  • ⚠ Balises <img> raw au lieu de next/image (12 instances)
  • ✗ Pas de content-visibility sur les listes longues (menu)
Reduire le nombre de polices Google Fonts P1
5 familles de polices sont chargees (Inter, Poppins, Montserrat, Indie Flower, Playfair Display). Chaque police ajoute ~20-40KB et bloque le rendu. La regle kb-performance.mdc recommande de minimiser les polices. Garder maximum 2-3 polices : une pour les titres, une pour le corps, une decorative si necessaire.
layout.tsx kb-performance.mdc LCP
Remplacer toutes les balises <img> par next/image P1
12 instances de <img> brutes trouvees (AnnouncementPopup, Footer, etc.). Cela bypasse l'optimisation automatique (lazy loading, formats modernes, responsive). Utiliser next/image avec les props sizes, width, height pour prevenir le CLS.
AnnouncementPopup.tsx Footer.tsx kb-performance.mdc
Ajouter content-visibility sur la liste du menu P2
La regle kb-server-fetching.mdc recommande content-visibility: auto; contain-intrinsic-size: 0 300px; sur les elements de liste longue (PizzaCard dans le menu). Cela accelere le rendu initial jusqu'a 10x pour les listes de 50+ items.
PizzaCard.tsx MenuPageClient.tsx kb-server-fetching.mdc
Lazy-loader Recharts dans l'admin P3
Le composant RevenueChart.tsx importe Recharts directement. La regle kb-async-bundle.mdc recommande un dynamic import avec ssr: false pour les librairies lourdes. Recharts ajoute ~200KB au bundle admin.
RevenueChart.tsx kb-async-bundle.mdc
Optimiser les imports Framer Motion P3
Verifier que les imports Framer Motion utilisent import { motion } from "framer-motion" et non le barrel import complet. Ajouter framer-motion dans optimizePackageImports de next.config.js pour le tree-shaking automatique.
next.config.js kb-async-bundle.mdc

5. Accessibilite (WCAG)

Conformite ARIA, semantique HTML, navigation au clavier, contrastes

  • ✓ aria-modal="true" et role="dialog" sur les modals
  • ✓ aria-live="polite" pour le suivi de commande
  • ✓ HTML semantique (main, nav, footer, section)

  • ✗ Contrastes de couleurs non valides (coral sur fond clair)
  • ⚠ Focus management incomplete sur le menu mobile
  • ⚠ Labels de formulaire pas tous associes explicitement
Ajouter un lien "Skip to content" P1
Aucun lien "Aller au contenu principal" n'est visible pour la navigation au clavier. C'est un critere WCAG 2.1 niveau A (2.4.1). Ajouter un <a href="#main-content"> visuellement cache sauf au focus, positionne avant le Header dans le layout.
layout.tsx LayoutSwitch.tsx WCAG 2.4.1
Corriger les contrastes de couleurs P1
La couleur corail (#E8614D) utilisee pour les CTAs et les liens sur fond clair risque de ne pas atteindre le ratio 4.5:1 exige par WCAG AA. La regle kb-mobile-responsive.mdc exige 4.5:1 minimum. Verifier avec un outil de contraste et ajuster les teintes si necessaire (ex: assombrir a #D4523F pour les textes).
globals.css tailwind.config.js WCAG 1.4.3
Ameliorer le focus trapping dans les modals P2
Les modals (PizzaOptionsModal, CartDrawer, Menu mobile) doivent trapper le focus clavier a l'interieur quand elles sont ouvertes. Implementer un hook useFocusTrap() ou utiliser @radix-ui/react-dialog qui gere cela nativement.
Modal.tsx PizzaOptionsModal.tsx WCAG 2.4.3
Associer tous les labels de formulaire P2
Dans CartDrawer, certains champs utilisent des placeholders au lieu de labels associes via htmlFor/id. Les lecteurs d'ecran ne peuvent pas annoncer correctement ces champs. Ajouter des <label> visibles ou utiliser aria-label.
CartDrawer.tsx WCAG 1.3.1
Ajouter aria-hidden aux icones decoratives P3
Certains SVG et icones (emojis, fleches decoratives) ne sont pas marques aria-hidden="true", ce qui pollue la lecture par les lecteurs d'ecran.

Aucune banniere de consentement n'est presente. Meme sans analytics, le localStorage est utilise (panier, session chatbot). La regle kb-cookie-consent.mdc exige vanilla-cookieconsent v3 avec Google Consent Mode v2 par defaut a "denied". Implementer le skill cookie-consent-setup complet.
